## 3. 新旧逻辑冲突清单（需优先消解）

1. structured draft 状态语义冲突：前端有 `rejected` 状态展示，但后端主链路仅见 `pending/applied` 写入与流转。  
2. 路由模型层级冲突：`docs/11.md` 目标是“模型池路由”，当前实现仍为 task->provider/model/fallback。  
3. 连续性上下文深度冲突：目标要求六类上下文稳定供给，但默认策略下 `window/recent` 可能缺席。  
4. 技能回写粒度冲突：目标要求技能驱动细粒度状态账本，当前多为通用 runtime key 写回。  
5. 场景编排强度冲突：目标是“场景感知编排闭环”，当前主要是关键词驱动与单次任务编排。

## 4. 下一步执行建议（按优先级）

1. P0：补齐 structured draft `rejected` 后端命令、状态机与测试，先消除前后端语义不一致。  
2. P0：先做“状态账本结构化 schema + 抽取写回”最小闭环，优先覆盖情绪/场景/关系三类。  
3. P0：将章节关键任务的 continuity pack 深度提升到可保障六类上下文完整。  
4. P0：设计模型池 DTO（先不大改 UI），打通 task route -> pool -> provider/model 的兼容链。  
5. P1：将确定性分区从 `step-08` 扩展至更关键的蓝图步骤，并升级 freeze 冲突判定策略。
