## [R-001] Review Trail 聚合化（chapter 维度）
- **日期**：2026-05-02
- **目标**：将 `Review Trail` 从“仅人工审查动作”升级为“五对象模型”所需的可追溯时间线基础（对齐 `Review Trail` 方向）。
- **范围**：
  - `src-tauri/src/services/review_trail_service.rs`
  - `docs/22.md`（本记录）
- **冲突描述**：
  - 旧逻辑：`get_review_trail` 仅查询 `user_review_actions`，无法体现 AI 生成链路与反馈闭环状态。
  - 新逻辑需求：chapter 轨迹应覆盖 AI 生成记录、自动推进结果、用户审查动作与反馈闭环结果，支持“为什么现在是这个结果”的追溯。
- **决策**：采用“新逻辑优先 + 渐进式迁移（兼容层）”
  - 保持既有 command 入参与返回 DTO 不变（兼容前端调用）。
  - 在 `entity_type = chapter` 时，聚合 `user_review_actions + ai_pipeline_runs + feedback_events`。
  - 非 chapter 查询保持旧行为，仅返回 `user_review_actions`。
- **实现**：
  - 重构 `ReviewTrailService.get_review_trail` 为聚合查询并统一时间倒序输出。
  - 新增聚合来源映射：
    - `ai_pipeline_runs` → `entity_type=ai_pipeline_run`，`action=ai_*`
    - `feedback_events` → `entity_type=feedback_event`，`action=feedback_*`
  - 保留旧结构：`ReviewTrailRecord` 字段不变，通过 `detail` 承载来源细节。
  - 新增单元测试 2 条：
    - chapter 维度聚合包含三类来源记录。
    - 非 chapter 维度保持旧逻辑（仅用户审查动作）。
- **验证**：
  - `cargo test --manifest-path src-tauri/Cargo.toml review_trail_service::tests` 通过（2 passed, 0 failed）。
- **风险**：
  - 当前 `feedback_events` 仅保存最终状态，历史状态迁移（open→ack→resolved）的完整事件序列仍不完整。
  - 前端“审查轨迹”标签文案目前主要针对 `approved/rejected/edited`，新增 `ai_* / feedback_*` 动作显示将走兜底文案。
- **下一步**：
  - 在不改接口的前提下，补充前端 `review action` 展示映射，提升可读性。
  - 评估是否为 `feedback_events` 增加状态迁移历史表，以支撑完整闭环复盘。

#### 高危问题

##### [H-01] 导出输出路径未限制在项目安全边界内
- **严重度**：高危
- **分类**：安全 / 数据
- **位置**：`src-tauri/src/services/export_service.rs:307`，`src-tauri/src/commands/export_commands.rs:8`
- **表现**：导出请求接受调用方传入的 `output_path`；相对路径直接 `project_root.join(path)`，绝对路径直接使用，未拒绝 `..`、Root/Prefix 组件，也未强制输出到 `exports/`。
- **影响**：若渲染层或命令调用被滥用，可将导出内容写到项目目录外，造成任意文件覆盖或敏感路径污染。
- **证据**：`write_export_output()` 与 `resolve_output_path()` 只负责拼接与建目录；项目已有安全工具 `resolve_project_relative_path()` 但导出路径未使用。
- **建议**：导出命令默认只允许项目相对路径且拒绝 `ParentDir`、Root、Prefix；MVP 阶段优先强制输出到 `exports/`，绝对路径仅允许来自可信文件选择流程。
- **修复成本**：0.5-1 天，需补充路径穿越单元测试。
- **是否为新旧冲突**：否。

##### [H-02] 上下文完整性门禁可能阻断新项目 AI 主闭环启动
- **严重度**：高危
- **分类**：业务 / 架构
- **位置**：`src-tauri/src/services/project_service.rs:97`，`src-tauri/src/services/ai_pipeline/continuity_pack.rs:259`，`src-tauri/src/services/ai_pipeline/orchestrator.rs:1192`
- **表现**：默认策略开启完整性门禁；基础必需层对所有任务要求 canon/state/promise 等上下文。空项目或早期项目执行蓝图、角色、设定生成时，这些层可能尚不存在。
- **影响**：可能阻断“项目 -> 蓝图 -> 章节写作”主闭环的 AI bootstrap 阶段。
- **证据**：静态审查显示 `compile_required_layers()` 对所有任务添加基础层；`collect_global_context_only()` 在无章节/资产时难以产生 canon/state/promise 层。
- **建议**：按任务类型拆分门禁；bootstrap 任务使用最小上下文要求，章节连续性任务继续硬阻断。
- **修复成本**：0.5-1 天，需补充空项目 AI bootstrap 与章节缺层阻断测试。
- **是否为新旧冲突**：是。
- **清理策略**：先完成新逻辑修正，再清理过宽门禁。
