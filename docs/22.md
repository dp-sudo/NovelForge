## 2. 对 `11.md` 的系统化理解（结论）

`11.md` 的核心不是“多做几个页面”，而是把系统从“单次 AI 生成器”升级为“持续生产系统”：

1. 用五对象模型管理权威层与执行层：`Constitution` / `Canon` / `State` / `Workspace` / `Review Trail`。
2. 用“晋升-回报-编译”取代“平铺同步”。
3. 用技能编排系统（workflow/capability/extractor/review/policy）驱动场景化生成。
4. 用确定性分区（冻结/承诺/探索）作为长期漂移防护。
5. 用状态账本回写把“写作表现能力”变成“连续性控制能力”。

## 4. 已确认的“旧逻辑 vs 新逻辑”冲突清单（必须重构）

### P0（优先处理）

1. `autoPersist` 兼容桥与 `persistMode` 显式策略并存，存在双语义风险。  
   - 影响：持久化行为不易推理，调用方可能继续依赖旧桥接。  
   - 位置：[`pipelineApi.ts`](/F:/NovelForge/src/api/pipelineApi.ts), [`orchestrator.rs`](/F:/NovelForge/src-tauri/src/services/ai_pipeline/orchestrator.rs)

2. Pipeline 核心规则散落在 `orchestrator.rs` 与 `task_handlers.rs`。  
   - 影响：冻结区守卫、持久化策略、状态回写策略无法独立验证。  
   - 位置：[`orchestrator.rs`](/F:/NovelForge/src-tauri/src/services/ai_pipeline/orchestrator.rs), [`task_handlers.rs`](/F:/NovelForge/src-tauri/src/services/ai_pipeline/task_handlers.rs)

3. 兼容命令仍开放在 `invoke_handler`。  
   - 影响：新逻辑可能被旧入口绕过。  
   - 位置：[`lib.rs`](/F:/NovelForge/src-tauri/src/lib.rs), [`settings_commands.rs`](/F:/NovelForge/src-tauri/src/commands/settings_commands.rs), [`ai_commands.rs`](/F:/NovelForge/src-tauri/src/commands/ai_commands.rs)

### P1（第二优先级）

1. 场景推断仍主要依赖关键词，未充分结合 `stateSummary` 与章节结构。  
   - 位置：[`orchestrator.rs`](/F:/NovelForge/src-tauri/src/services/ai_pipeline/orchestrator.rs)

2. 状态回写 payload 以审计元数据为主，缺少“情绪/外观/服饰/关系温度”等结构化字段规范。  
   - 位置：[`task_handlers.rs`](/F:/NovelForge/src-tauri/src/services/ai_pipeline/task_handlers.rs)

3. 前端三个核心页面过大（Blueprint/Editor/Settings），新旧逻辑耦合。  
   - 位置：[`BlueprintPage.tsx`](/F:/NovelForge/src/pages/Blueprint/BlueprintPage.tsx), [`EditorPage.tsx`](/F:/NovelForge/src/pages/Editor/EditorPage.tsx), [`SettingsPage.tsx`](/F:/NovelForge/src/pages/Settings/SettingsPage.tsx)

## 5. 当前阶段结论

1. 新逻辑主干已成立：五对象、确定性分区、技能编排、状态账本、Pipeline 事件流都已实装并有契约测试支撑。  
2. 当前最大问题不是“功能缺失”，而是“实现过于集中，规则分散，旧桥接仍在”，导致维护复杂度高。  
3. 后续必须执行“新逻辑优先、旧逻辑收束”重构：先把策略规则模块化，再逐步下线兼容桥。
