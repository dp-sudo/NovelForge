# NovelForge AI 生产系统与技能编排设计方案

日期：2026-04-30  
状态：设计确认稿  
范围：目标方案 + 分阶段迁移方案  
约束：以当前代码真实行为为基线，不以旧文档的理想描述为基线

## 1. 文档目标

本文档定义 NovelForge 后续的核心生产机制，重点解决 4 个问题：

1. `AI 如何作为主流生产力持续推进蓝图、资产、状态和正文生成`
2. `项目数据如何支撑长篇连载的连续性、严谨性和可维护性`
3. `用户如何以较低干预成本完成方向指导、审查、修订、确认`
4. `技能系统如何从平铺 Prompt 模板升级为可编排的生产能力系统`

本文档不讨论云协作、多人实时协同、插件市场，不覆盖与 MVP 主闭环无关的泛化架构。

## 2. 当前代码基线

本节只记录已经能从仓库代码中直接验证的事实。

### 2.1 项目级数据

当前项目根目录包含：

- `project.json`
- `database/project.sqlite`
- `manuscript/chapters/*.md`
- `manuscript/drafts/*.md`
- `blueprint/`
- `assets/`
- `exports/`

项目创建由 `ProjectService` 完成，`project.json` 只在创建时写入基础元数据和默认设置。项目运行期的大部分正式数据仍以项目库为主，正文则落 Markdown 文件。

### 2.2 第一层：蓝图规划层的当前实现

当前“创作蓝图”存于项目库 `blueprint_steps` 表，每个步骤一条记录，`content` 存 JSON 字符串。前端通过 `parseBlueprintContent/serializeBlueprintContent` 把它映射为结构化表单。

当前蓝图共有 8 个步骤：

1. 灵感定锚
2. 类型策略
3. 故事母题
4. 角色工坊
5. 世界规则
6. 名词锁定
7. 剧情骨架
8. 章节路线

蓝图页面已经是结构化编辑器，不是自由富文本页；但它仍然是“步骤 JSON 表单层”，不是正式实体层。

### 2.3 第二层：正式资产层的当前实现

当前项目库已经有正式业务表，主要包括：

- `characters`
- `world_rules`
- `glossary_terms`
- `plot_nodes`
- `narrative_obligations`
- `character_relationships`
- `chapters`
- `chapter_links`

这些表已经是正式资产层，而不是草稿层。当前模块页的 CRUD 基本都直接作用于这些正式表。

### 2.4 第三层：正文执行层的当前实现

章节正文不存数据库，而是存放在 `manuscript/chapters/*.md`。数据库中保存章节元数据、摘要、目标字数、状态、关联等。

这意味着：

- 正式章节写作载体是文件系统
- 数据库承担元数据、关联和结构化资产职责

### 2.5 编辑器侧的现有候选池

当前项目已经存在一套“先抽取候选，再人工确认入库”的机制，但它只在编辑器链路中成立。

相关表：

- `structured_draft_batches`
- `structured_draft_items`

当前可抽取的主要内容：

- 资产候选
- 关系草案
- 出场草案
- 场景草案

用户可通过 `apply_structured_draft` 把候选正式落库。

### 2.6 应用级 AI 配置的当前实现

当前 Provider、模型、任务路由并不属于项目库，而属于应用级数据库 `~/.novelforge/novelforge.db`。

应用级数据库当前承载：

- `llm_providers`
- `llm_models`
- `llm_task_routes`
- `llm_model_registry_state`
- `app_settings`

这意味着当前系统中：

- API Key 和 Provider 配置是应用级
- 任务路由是应用级
- 项目级几乎没有独立的 AI 策略层

### 2.7 项目级 AI 配置的当前实现

当前项目级真正稳定存在的 AI 相关配置只有 `writing_style`。

它的字段包括：

- `languageStyle`
- `descriptionDensity`
- `dialogueRatio`
- `sentenceRhythm`
- `atmosphere`
- `psychologicalDepth`

当前这套写作风格参数会注入 Prompt，但它本质上只是一个文风向量，不是项目级 AI 策略层。

### 2.8 技能系统的当前实现

当前技能系统本质上是：

`Markdown 技能文件 + frontmatter manifest + Prompt 模板正文`

当前 manifest 已有字段：

- `id`
- `name`
- `description`
- `category`
- `tags`
- `inputSchema`
- `outputSchema`
- `requiresUserConfirmation`
- `writesToProject`
- `taskRoute`（可选）

但运行期真正有效的能力仍然较窄：

1. 技能内容主要作为 Prompt 模板使用
2. 技能列表主要用于 UI 展示和编辑
3. 技能 `taskRoute` override 在主 pipeline 中没有真正打通

### 2.9 一键全书生成的当前实现

当前 `bookPipelineApi` 不是“只生成蓝图”，而是：

1. 先生成 8 个蓝图步骤
2. 再直接生成并持久化：
   - 角色
   - 世界规则
   - 剧情节点
   - 名词
   - 叙事义务
   - 可选章节计划

这意味着当前实现已经跨层：

`蓝图生成` + `正式资产落库`

混在同一条主链路中。

### 2.10 Prompt 编译的当前实现

当前 `PromptResolver::context_to_string()` 注入的上下文仍然偏薄，主要是：

- 作品名、题材、POV
- 写作风格向量
- 已完成蓝图步骤的截断预览
- 当前章节摘要
- 前章摘要
- 剧情节点标题
- 角色名
- 关系边描述
- 世界规则短预览

当前缺失的关键上下文包括：

- 角色当前状态
- 角色当前已知信息边界
- 当前卷目标
- 当前窗口计划
- 未兑现叙事义务
- 当前活跃冲突
- 当前场景状态
- 穿着/外貌/情绪等可连续性追踪数据

结论：当前系统能做“上下文拼接”，还不能做真正的“上下文编译”。

## 3. 当前系统的关键缺口

结合现有实现，当前系统要支撑长篇连载，主要缺 6 个能力。

### 3.1 缺项目级 AI 策略层

当前项目级只有文风设置，没有项目级 AI 生产策略，例如：

- 哪些任务默认自动推进
- 哪些任务默认进入待审
- 哪些技能包常驻启用
- 当前项目对连贯性审查的严格度
- 章节生产时默认启用哪些能力

### 3.2 缺动态故事状态层

当前正式资产层能回答“故事里有什么”，但不能稳定回答“故事现在变成了什么”。

例如当前缺少统一承载：

- 人物当前身体/精神/情绪状态
- 当前穿着、外貌变化、伪装状态
- 当前场景环境状态
- 势力格局变化
- 主线/卷线推进状态
- 未兑现伏笔与义务的当前状态

### 3.3 缺上下文编译器

当前写作前拿到的上下文是文本拼接结果，而不是面向当前任务的结构化连续性包。

这会直接导致长篇后期问题：

- 人物设定不漂，但行为漂
- 世界规则不漂，但当前状态漂
- 长线还在，近期推进发散

### 3.4 缺蓝图与正式资产的权威分层

当前蓝图步骤和正式资产在一键全书生成中被混写，导致：

- 用户尚未确认总纲，正式资产已被写入
- 蓝图与正式资产可能从第一天起就分叉

### 3.5 缺技能编排层

当前技能系统更像“可编辑 Prompt 模板仓库”，而不是生产能力编排系统。

这意味着：

- 技能之间没有明确组合机制
- 技能没有场景触发条件
- 技能没有与数据层的回写契约
- 技能没有自动化分级策略

### 3.6 缺系统级审查轨迹

当前有 `ai_requests` 和 pipeline 事件，但还没有真正表达：

- AI 生成了什么
- 自动推进了什么
- 用户改了什么
- 用户否决了什么
- 哪些结果被升级为正式事实

这会削弱回退、纠偏和复盘能力。

## 4. 总体目标

目标不是把系统改造成“人工逐项审批工具”，也不是改造成“AI 全自动写书器”。

目标定义如下：

> NovelForge 是一个由项目内 AI 策略驱动、由 AI 作为主流推进生产力、由用户保留方向指导、审查、细致修订和关键确认权的长篇小说生产系统。

三个硬原则：

1. `AI 主导推进`
2. `用户保留审查、修订、确认权`
3. `正式事实、动态状态、章节生成必须纳入连续性闭环`

---

## 5. 目标系统：五对象模型

为避免“页面等于数据层”的误解，目标系统按 5 个核心对象定义。

### 5.1 Story Constitution：故事宪法

定位：第一层的升级定义。  
职责：保存最稳定、最不该漂移的长期约束。

保存内容：

- 核心命题
- 类型策略
- 世界底层不变量
- 主角/反派终局方向
- 主线终局方向
- 文风红线
- 不可违背规则

它回答的问题是：

`这本书绝对不能偏到哪里去`

### 5.2 Canon Registry：正式资产库

定位：当前第二层正式资产层的统一定义。  
职责：保存已经正式成立的长期事实。

保存内容：

- 角色
- 地点
- 组织
- 术语
- 世界规则
- 剧情节点
- 叙事义务
- 章节计划元数据
- 正式关系

它回答的问题是：

`这个故事里正式存在什么`

### 5.3 State Ledger：动态状态账本

定位：新增的关键层。  
职责：保存故事推进后的当前状态。

保存内容：

- 人物当前状态
- 人物当前情绪与关系温度
- 人物当前已知信息边界
- 当前穿着/外貌/伤病/伪装变化
- 场景当前状态
- 势力格局当前状态
- 道具/资源当前归属
- 主线与卷线当前推进状态
- 未兑现伏笔与义务
- 当前窗口写作状态

它回答的问题是：

`故事现在推进到了哪里`

### 5.4 Execution Workspace：执行工作区

定位：写作前临时编译出的任务工作区。  
职责：把多层数据编译成当前任务真正可用的连续性包。

包含内容：

- 故事宪法约束
- 当前卷目标
- 当前窗口计划
- 当前章节目标
- 当前相关正式资产
- 当前相关动态状态
- 近期承接摘要
- 风险提示

它回答的问题是：

`这一次 AI 应该在什么边界内生成`

### 5.5 Review Trail：审查轨迹

定位：跨层审计对象。  
职责：记录 AI 推进与人工干预历史。

记录内容：

- 生成来源
- 自动推进动作
- 用户修订动作
- 确认/否决动作
- 晋升路径
- 回退依据

它回答的问题是：

`当前结果为什么会变成现在这样`

补充边界：

- `Review Trail` 在本方案中首先是逻辑对象，不默认新增一套独立持久化层
- 当前优先复用和扩展已有 `ai_pipeline_runs`、`PipelineAuditStore` 与 pipeline 事件流
- 只有当现有审计载体无法承载用户修订轨迹或晋升轨迹时，才新增更细粒度的审计表

逻辑对象与当前实现的映射优先级如下：

| 逻辑对象 | 当前可复用实现 | 目标演进 |
| --- | --- | --- |
| AI 任务运行轨迹 | `ai_pipeline_runs` + `PipelineAuditStore` | 继续承载 task/phase/status/ui_action 级审计 |
| 运行期阶段推进 | orchestrator phase + `ai:pipeline:event` 事件 | 继续承载 generate/persist/review 等阶段信号 |
| 用户修订/确认/否决轨迹 | 当前缺统一承载 | 优先扩展现有审计存储；仅在不足时新增细粒度表 |
| 晋升/回退依据 | 当前散落在正式层与候选层动作里 | 统一挂入 Review Trail 逻辑视图 |

## 6. 系统机制：不是“同步”，而是三种动作

后续设计中，核心动作不再只用“同步”描述，而用 3 个更准确的概念。

### 6.1 晋升（Promotion）

从低权威结果升级为高权威结果。

例如：

- AI 草案 -> 正式蓝图
- 蓝图内容 -> 正式资产
- 抽取候选 -> 正式角色/关系/场景
- 状态草案 -> 正式状态账本记录

### 6.2 回报（Feedback）

从下层运行结果向上层返回摘要，而不是全文回灌。

例如：

- 第二层新增多个新势力，不直接把所有势力明细塞回蓝图
- 第一层只收到“本卷势力结构升级，需要补卷级规划”

### 6.3 编译（Compile）

写作前，系统按任务把多层数据编译成连续性包，而不是简单拼接文本。

这是后续高质量长篇生成的核心能力。

## 7. 权威级别设计

系统中的数据必须有明确权威级别。

### 7.1 宪法权威

高稳定，低频变更，一旦变化影响全书。

例子：

- 世界底层规则
- 作品主命题
- 主角终局成长弧

### 7.2 正式事实权威

已经正式成立，但允许增量扩展。

例子：

- 角色核心身份
- 正式术语定义
- 正式组织立场
- 正式剧情节点

### 7.3 当前状态权威

高频变更，服务后续承接。

例子：

- 角色当前受伤
- 当前情绪状态
- 当前场景控制权
- 当前秘密是否暴露

### 7.4 草案权威

允许 AI 先行推进，但不直接等于正式事实。

例子：

- 章节计划草稿
- 新角色候选
- 新关系候选
- 状态更新草案

## 8. 第一层设计：从“蓝图表单”升级为“滚动式故事宪法”

### 8.1 第一层仍复用蓝图页面作为入口

不新增一套完全独立的“宪法编辑器”。  
第一阶段继续复用 `BlueprintPage` 作为第一层入口。

### 8.2 规划粒度升级为三层

第一层后续应支持：

1. `总书级`
2. `卷级`
3. `窗口级`

#### 总书级

保存：

- 核心命题
- 底层规则
- 长线终局方向

#### 卷级

保存：

- 每卷目标
- 每卷核心冲突
- 每卷转折与高潮

#### 窗口级

保存：

- 未来 10-20 章近期路线
- 当前活跃人物线
- 当前窗口必须兑现的承诺

### 8.3 增加三种确定性分区

为支撑长篇连载，第一层内容还需要分成 3 个区。

#### 冻结区

已成正式事实，不允许 AI 自行改写。

#### 承诺区

已经规划或埋下，后续必须兑现。

#### 探索区

允许 AI 在边界内试探扩展，供用户挑选。

### 8.4 第一层的长期职责

第一层只维护三类信息：

1. 稳定不变量
2. 长线规划
3. 阶段性摘要

第一层不应持续吸纳全量新增角色、场景、组织明细。

## 9. 第二层设计：正式资产库 + 动态状态账本

### 9.1 第二层A：正式资产库

当前现有表大体可复用：

- `characters`
- `world_rules`
- `glossary_terms`
- `plot_nodes`
- `narrative_obligations`
- `character_relationships`
- `chapters`

但需要补足统一元属性。

建议为资产补充 3 个统一维度：

1. `稳定度`
   - 临时
   - 重要
   - 核心

2. `作用域`
   - 单章
   - 窗口级
   - 卷级
   - 全书级

3. `热度`
   - 最近活跃
   - 当前活跃
   - 已休眠
   - 已结束

这样后续上下文编译时才能精确挑选资产，而不是全量喂入。

### 9.2 第二层B：动态状态账本

当前仓库没有统一状态层，因此后续需要新增。

为控制复杂度，MVP 后续推荐先采用一个通用状态表，而不是一开始拆多张细表。

建议新增：

`story_state`

建议字段：

- `id`
- `project_id`
- `subject_type`
  - `character`
  - `relationship`
  - `location`
  - `organization`
  - `world`
  - `plot`
  - `window`
- `subject_id`
- `scope`
  - `chapter`
  - `window`
  - `volume`
  - `global`
- `state_kind`
  - `emotion`
  - `body`
  - `appearance`
  - `clothing`
  - `location`
  - `knowledge`
  - `conflict`
  - `promise`
  - `environment`
  - `ownership`
  - `progress`
- `payload_json`
- `source_chapter_id`
- `status`
  - `active`
  - `superseded`
  - `resolved`
- `created_at`
- `updated_at`

这张表先解决“统一承载动态状态”的问题，不提前过度拆分。

边界约束：

- `story_state` 是状态账本，不替代 `characters`、`plot_nodes`、`narrative_obligations` 等 typed 正式资产表
- 当前已有半动态字段继续保留在原表中，状态账本负责补充“当前状态快照 + 增量变化”
- 后续如发现高频状态需要独立索引或独立查询，再从账本中拆出专门结构，而不是一开始就把 typed 资产层并入状态层

### 9.3 第二层的当前半动态字段应被保留并纳入账本体系

当前已有一些半动态字段：

- `characters.arc_stage`
- `plot_nodes.status`
- `plot_nodes.emotional_curve`
- `narrative_obligations.payoff_status`

这些字段不应被废弃。  
后续应将其视为：

- 正式资产的稳定摘要字段
- 动态账本的压缩投影字段

即：

- 正式表保留“当前摘要值”
- 账本保存“变化过程”

## 10. 写作连续性：Continuity Pack 上下文编译器

### 10.1 为什么当前上下文不够

当前 Prompt 注入主要是：

- 蓝图预览
- 角色名
- 世界规则预览
- 前章摘要

这不够支撑长篇，因为它缺乏：

- 当前状态
- 当前承诺
- 当前窗口目标
- 当前场景边界

### 10.2 目标：写前编译 Continuity Pack

生成章节前，系统必须按任务编译一份 `Continuity Pack`。

建议最小内容：

1. `Constitution Context`
   - 不可漂移约束

2. `Canon Context`
   - 当前章相关正式资产

3. `Lexicon Policy Context`
   - 锁定术语、禁用词、命名约束

4. `State Context`
   - 当前活跃人物/关系/环境状态

5. `Promise Context`
   - 当前必须兑现的伏笔、义务、承诺

6. `Window Plan Context`
   - 当前窗口目标与节奏要求

7. `Recent Continuity Context`
   - 最近 1-3 章承接摘要

补充约束：

- `locked_terms`、`banned_terms` 不是 Canon Context 的隐含部分，而是 Continuity Pack 的显式一等输入
- 任何章节生成、章节续写、模块补全、蓝图改写任务，只要涉及命名与正文输出，都应显式消费这类约束
- 这一步是对当前 `collect_global_context()` 已有数据的升级利用，不是凭空新增概念

### 10.3 章节生成工作流

章节生成不应一步直接出终稿。

建议链路：

1. 章节计划
2. 场景分解
3. 正文草稿
4. 一致性审查
5. 必要润色
6. 用户确认采用
7. 写后抽取新增资产与状态

这样 AI 仍是主推进者，但不会在长篇中盲推失控。

## 11. AI 与人工的权限边界

### 11.1 总原则

AI 是主流生产力。  
用户不是主要生产者，但必须拥有：

- 方向输入权
- 审查权
- 细修权
- 关键确认权
- 否决权

### 11.2 分级自动化

不是所有动作都要事前人工确认。  
需要按风险分级。

#### 自动档

AI 可自动推进，用户事后审看即可。

适用：

- 低风险术语补充
- 时间线整理
- 状态摘要初稿
- 非核心候选抽取

#### 监督档

AI 自动推进，但结果进入“待审查”状态。

适用：

- 新关系边
- 新支线节点
- 场景环境状态更新
- 卷级规划调整建议

#### 确认档

AI 只能生成候选，用户必须确认后才生效。

适用：

- 总书级规划改写
- 核心角色重定义
- 核心世界规则变更
- 正文章节终稿采用
- 删除或覆盖正式事实

### 11.3 用户的实际角色

用户应被设计成：

- 低频干预
- 关键纠偏
- 最终把关

而不是逐条手工操作员。

## 12. 技能系统设计

### 12.1 当前技能系统的问题

当前技能系统有 3 个明显问题：

1. 本质仍是平铺 Prompt 模板
2. 技能 manifest 的很多字段只停留在展示层
3. 技能之间没有工作流编排能力

### 12.2 目标：保留 Markdown，可升级执行语义

不推翻 Markdown 技能文件机制。  
继续保留“Markdown 可编辑技能”作为内容源，但扩展 manifest 的执行语义。

### 12.3 技能五分类

后续技能按 5 类组织。

#### 1. Workflow Skills

面向用户的主任务入口。

例如：

- 生成蓝图步骤
- 生成章节计划
- 生成章节草稿
- 续写章节
- 改写选区
- 一致性扫描

#### 2. Capability Skills

可注入的细分能力，不必都直接暴露为按钮。

例如：

- 人物肢体动作
- 人物外貌与服装变化
- 情绪推进
- 场景环境渲染
- 背景信息渗透
- 对话声纹控制
- 战斗规则兑现

#### 3. Extractor Skills

写后抽取新增事实与状态。

例如：

- 新角色抽取
- 新关系抽取
- 新场景抽取
- 新情绪变化抽取
- 新穿着/外貌变化抽取

#### 4. Review Skills

负责发现连续性、逻辑性、风格性问题。

例如：

- 人物状态连续性审查
- 情绪弧线审查
- 场景环境连续性审查
- 规则突破审查

#### 5. Policy Skills

常驻策略与约束技能。

例如：

- 文风约束
- 术语锁定
- 禁用表达
- POV 限制
- 底层规则守卫

### 12.4 核心能力包设计

结合当前需求，建议定义 4 个核心能力包。

#### 人物表现包

包含：

- 肢体动作
- 面部细节
- 外貌描写
- 服装变化
- 对话声纹
- 情绪外化

#### 情绪推进包

包含：

- 当前情绪识别
- 情绪升级/回落
- 压抑与爆发节奏
- 情绪与关系联动
- 内心深度控制

#### 场景环境包

包含：

- 空间结构
- 光线/声音/气味/温度
- 背景信息渗透
- 环境与情绪联动
- 场景过渡控制

#### 规则兑现包

包含：

- 能力边界
- 代价机制
- 战斗动作逻辑
- 结果后果追踪
- 世界规则校验

### 12.5 技能不是只写文本，还要回写数据

这些能力不能只改善文笔，还要参与状态维护。

例如：

#### 外貌/服装/动作技能

除了生成描写，还应回写：

- 当前外貌状态
- 当前服装状态
- 当前受伤/疲劳状态

#### 情绪技能

除了生成心理或行为表达，还应回写：

- 当前主情绪
- 情绪触发源
- 与关键关系的温度变化

#### 场景环境技能

除了写环境，还应回写：

- 当前场景状态
- 天气/时间/危险等级
- 环境是否发生变化

### 12.6 技能系统升级前提

当前技能系统还不具备直接承载高级编排字段的落地条件，原因有 4 个：

1. `create_skill` 当前只能创建最小 manifest
2. `update_skill` 当前只能更新 body，不能更新 manifest 元数据
3. 技能管理 UI 当前主要是列表浏览和正文编辑，不是 manifest 元数据编辑器
4. 运行期当前主要消费 prompt 模板，连现有 `task_route` override 都尚未在主 pipeline 中完全打通

因此技能系统升级必须先拆成 4 条迁移前提：

1. `manifest schema 迁移`
   - 扩展 `SkillManifest` 结构
   - 明确哪些字段属于 phase 1 必备，哪些属于后续高级字段

2. `命令与 API 迁移`
   - 扩展 `create_skill/update_skill`，允许编辑 manifest 元数据
   - 同步前后端 DTO，不再只允许改 body

3. `技能管理 UI 迁移`
   - 把设置页中的技能管理从“模板正文编辑器”升级为“manifest + body 双编辑器”
   - 为分类、常驻、自动化档位、触发条件等字段提供编辑入口

4. `运行期消费点迁移`
   - 明确哪些 manifest 字段在 prompt 解析阶段生效
   - 明确哪些字段在 route 解析阶段生效
   - 明确哪些字段在上下文编译、状态回写、审查阶段生效

只有在这 4 条前提成立后，后面的 bundle、trigger、stateWrites 等高级字段才有实现基础。

### 12.7 技能 manifest 的目标扩展字段

以下字段属于目标 schema 演进方向，不是当前阶段立即生效的默认能力。

在保留现有字段基础上，建议增加：

- `skillClass`
  - `workflow`
  - `capability`
  - `extractor`
  - `review`
  - `policy`
- `bundleIds`
- `alwaysOn`
- `triggerConditions`
- `requiredContexts`
- `stateWrites`
- `automationTier`
  - `auto`
  - `supervised`
  - `confirm`
- `sceneTags`
- `affectsLayers`

### 12.8 当前 skill route override 的处理原则

当前 manifest 虽然有 `task_route`，但主 pipeline 实际没有把 skill registry 传进路由解析链路。

因此目标方案必须明确：

1. 要么正式废弃该字段
2. 要么把它真正打通到主 pipeline

本方案选择第 2 种：

- 允许 workflow skill 或 bundle 声明路由偏好
- 但不直接保存 API Key
- 最终仍引用应用级 Provider/Model 配置

## 13. 项目级 AI 策略层设计

### 13.1 为什么需要新层

当前项目级只有 `writing_style`，不够表达：

- 自动化等级偏好
- 技能包启用偏好
- 审查严格度
- 长篇连续性策略

### 13.2 目标：新增项目级 AI Strategy Profile

建议新增项目级配置对象：

`ai_strategy_profile`

建议最小字段：

- `automation_default`
- `review_strictness`
- `default_workflow_stack`
- `always_on_policy_skills`
- `default_capability_bundles`
- `state_write_policy`
- `continuity_pack_depth`
- `chapter_generation_mode`
- `window_planning_horizon`

推荐保存位置：

- 运行期单一权威源：项目库
- 物理载体优先采用 `projects.ai_strategy_profile` JSON 列，保持与当前 `projects.writing_style` 的模式一致
- `project.json` 不再承担运行期真相源职责

### 13.3 项目级 AI 策略的权威源定义

当前代码里已经存在一个遗留双源问题：

- 项目创建时，`writing_style` 同时写入 `project.json` 和 `projects.writing_style`
- 项目运行期读取和保存则只走数据库
- 前端 `ProjectJson` 类型也已经不再把 `writing_style` 作为稳定运行期字段暴露

如果不先解决这个问题，新增 `ai_strategy_profile` 只会继续制造第二套分叉。

本方案明确采用以下规则：

1. `项目运行期真相源 = project.sqlite`
   - `writing_style`
   - 后续 `ai_strategy_profile`
   - 其他项目级 AI 策略偏好
   都以项目库为唯一运行期权威源

2. `project.json = 冷启动/导入导出/备份元数据`
   - 允许保留项目基础元信息
   - 允许保留创建时默认值快照
   - 不承担运行期配置读写真相源职责

3. `不再引入新的隐式双写`
   - 后续保存项目级 AI 策略时，不默认双写 `project.json`
   - 如需要生成可携带快照，应通过显式导出/同步动作完成，而不是在日常保存路径中隐式双写

### 13.4 与应用级配置的边界

#### 应用级继续保存

- Provider 配置
- API Key
- 模型注册表
- 全局任务路由

#### 项目级新增保存

- 当前项目的 AI 策略偏好
- 技能启用策略
- 自动化等级偏好
- 长篇连续性与审查偏好

这样可以满足“项目内配置 AI 推进策略”，同时不把密钥写入项目目录，也不会再复制出新的运行期双源。

## 14. 设置页的信息架构调整

### 14.1 保留现有页签，但重新定义职责

#### 模型配置

应用级 Provider、模型、能力注册。

#### 任务路由

应用级 `taskType -> provider/model` 路由。  
后续补充：

- 自动化默认级别

从“平铺技能列表”升级为：

- 技能分类

- 是否常驻

继续作为项目级文风向量，但不再单独承担全部项目 AI 策略。

#### 新增：AI 策略

### 14.2 写作风格与技能包联动

它还应影响：

- 人物表现包
- 情绪推进包
- 场景环境包
- 背景渗透策略

这样文风才真正统一。

## 15. 一键全书生成的目标链路

### 15.1 当前问题

当前一键全书生成会同时：

- 写蓝图
- 写正式资产

这违反了权威分层。

### 15.2 目标链路

建议拆成两个阶段。

#### 阶段一：生成第一层

仅生成：

- 蓝图步骤
- 总书/卷级/窗口级初稿

#### 阶段二：晋升到第二层

由 AI 主动拆解并写入：

- 正式角色
- 正式世界规则
- 正式名词
- 正式剧情节点
- 正式叙事义务
- 正式章节计划

但按自动化等级分为：

- 自动档
- 监督档
- 确认档

### 15.3 用户在这条链路中的位置

用户不需要手工逐条执行，但必须：

- 能看到结果摘要
- 能修订关键结果
- 能否决重大晋升

### 16.1 新章节写后必须有两类回写

#### 回写正式资产库

新增：

- 新角色
- 新地点
- 新术语
- 新关系
- 新剧情节点

#### 回写动态状态账本

更新：

- 情绪变化
- 外貌/服装变化
- 伤病/体力变化
- 场景环境变化
- 主线推进变化
- 伏笔兑现状态变化

### 16.2 第一层只接收摘要回报

第一层不接收全量明细，而只接收：

- 当前卷推进情况
- 关键变量新增情况
- 当前窗口是否偏航
- 是否需要调整卷级规划

### 16.3 时间线和关系图的定位

时间线和关系图仍然定位为：

- 第二层数据的派生审阅层
- 用于发现缺口、冲突、可晋升信息

不是一级事实源。

## 17. 分阶段迁移方案

### 阶段一：收口当前边界

目标：

- 明确第一层、第二层、第三层边界
- 停止蓝图和正式资产的混写

动作：

1. 先盘点所有 `autoPersist: true` AI 入口，而不是只看 `bookPipelineApi`
2. 入口清单至少覆盖：
   - `blueprint.generate_step`
   - `bookPipelineApi` 的阶段编排
   - 角色 / 世界 / 名词 / 剧情 / 叙事义务页面 AI 入口
   - 时间线 / 关系图页面 AI 入口
   - 一致性检查等审阅型 AI 入口
3. 按入口类型重新归类为：
   - 第一层草案推进
   - 第二层正式资产晋升
   - 派生审阅层自动更新
4. `一键全书生成` 收口为第一层生成优先，不再作为唯一需要治理的跨层入口
5. 蓝图页增加显式“确认并同步/晋升”动作
6. 模块页显示“来源于蓝图/来源于章节抽取”

### 阶段二：补项目级 AI 策略层

目标：

- 让项目级 AI 生产策略有稳定保存位置

动作：

1. 先定义项目级运行期真相源以项目库为准
2. 消除 `writing_style` 的遗留双源语义，明确 `project.json` 只作冷启动/备份元数据
3. 在项目库新增 `ai_strategy_profile`
4. 写作风格继续保留，并并入项目级策略读取语义
5. 设置页新增 AI 策略入口

### 阶段三：建立动态状态账本

目标：

- 解决长篇后期状态漂移

动作：

1. 新增 `story_state`
2. 章节确认后写状态更新
3. 写前注入状态层
4. 保持状态账本与 typed 资产层并存，不做替代式迁移

### 阶段四：建立 Continuity Pack 编译器

目标：

- 从“上下文拼接”升级为“上下文编译”

动作：

1. 明确定义 7 类上下文
2. 按任务选择上下文深度
3. 统一写前编译入口
4. 把 `locked_terms`、`banned_terms`、命名约束提升为显式输入

### 阶段五：升级技能系统

目标：

- 从平铺模板升级为可编排能力系统

动作：

1. 先做 manifest schema 迁移，而不是直接假设新字段可用
2. 扩展 `create_skill/update_skill` 与前后端 DTO，允许编辑 manifest 元数据
3. 升级技能管理 UI，提供 manifest + body 双编辑能力
4. 明确运行期消费点：prompt、route、context compile、review、state write
5. 在上述前提成立后，再增加技能五分类、能力包、触发条件、回写声明
6. 真正打通 skill route override 或用新配置替代

### 阶段六：完善长篇滚动规划

目标：

- 让第一层真正具备长篇连载规划能力

动作：

1. 引入总书级/卷级/窗口级结构
2. 引入冻结区/承诺区/探索区
3. 让窗口规划与章节生成闭环

## 18. 非目标

本方案当前不包括：

1. 多人协作
2. 云同步
3. 跨项目共享技能市场
4. 全自动无人工确认写完全书
5. 高频实时双向同步所有层

## 19. 方案落地后的验证标准

设计方案落地后，至少满足以下可验证目标：

1. `一键全书生成` 不再默认跨层混写蓝图与正式资产
2. 章节生成前可明确输出一份 Continuity Pack
3. 章节生成后可自动回写新增资产与新增状态
4. 项目级 AI 策略与应用级 Provider/Key 边界清晰
5. 技能系统可区分 workflow/capability/extractor/review/policy
6. 人物动作、着装、外貌、情绪、场景环境等能力可以作为能力包参与生产
7. 长篇连载中，后续章节可读到明确的当前状态，而不是只读静态设定

## 20. 最终结论

NovelForge 后续不应继续朝“页面越来越多、Prompt 越来越长”的方向演进。  
更合理的方向是：

1. 用 `Story Constitution` 管方向
2. 用 `Canon Registry` 管正式事实
3. 用 `State Ledger` 管当前状态
4. 用 `Execution Workspace` 管每次生成前的上下文编译
5. 用 `Review Trail` 管 AI 推进与人工纠偏的历史
6. 用 `Skill Orchestration` 管人物、情绪、场景、规则等生产能力的组合与回写

这样系统的主逻辑才会从“AI 生成按钮集合”升级为“AI 主导推进、用户监督修订、长篇可持续生产”的完整生产机制。
