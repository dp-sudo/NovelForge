- 工作区：`F:\NovelForge`
- 环境：Windows 10 (22H2)，PowerShell 优先
- 当前阶段：MVP 主闭环实现与验收
- 核心参考文档：
  - `docs/novel_workbench_windows_lifecycle_blueprint_v_1.md`
  - `docs/novelforge_llm_provider_integration_spec_v_1.md`

## 2. 非协商规则（必须遵守）
1. 不编造路径、接口、命令输出、测试结果、提交记录。
2. 先读实现再改文档；文档必须反映“已实现行为”，不是理想设计。
3. 仅改请求范围内内容，不做顺手重构。
4. 未验证前禁止宣称“完成/通过”。
5. 删除或移动文件时，仅操作明确目标。
6. 若存在关键歧义且会改变输出，先提问再实现。

## 3. 编码前先思考
不要假设，不要隐藏困惑，明确揭示权衡。

在着手实现前必须执行：
1. 明确陈述假设；如果不确定，先提问。
2. 若存在多种解释，逐一列出并说明取舍；禁止默不作声自行选择。
3. 若存在更简单方案，必须指出；在理由充分时应明确反对过度设计。
4. 若有不清楚之处，立即停止实现，指出困惑点并提问。

## 4. 简单至上
用最少代码解决问题，杜绝推测性代码。

1. 不添加未要求功能。
2. 不为仅使用一次的代码建立抽象。
3. 不添加未被要求的灵活性或可配置性。
4. 不对不可能发生的场景增加错误处理。
5. 如果 200 行可用 50 行完成，重写为更简单方案。
6. 每次实现前自检：资深工程师是否会认为过度复杂；若是，则继续简化。

## 5. 外科手术式修改
只改必须改的部分，只清理自己造成的混乱。

编辑现有代码时：
1. 不改进相邻无关代码、注释或格式。
2. 不重构没有问题的区域。
3. 遵循现有代码风格，即便个人偏好不同。
4. 发现不相关死代码时可指出，但不得擅自删除。

当修改产生孤立项时：
1. 删除因本次改动变成未使用的导入、变量、函数。
2. 除非明确要求，不删除原先已存在的死代码。

检验标准：
- 每一行改动都必须能直接追溯到用户需求。

## 6. MVP 范围边界
### 6.1 必做（主闭环）
- 新建项目 / 打开项目 / 最近项目
- 创作蓝图
- 角色 / 设定 / 主线基础能力
- 章节写作（保存、自动保存、恢复）
- AI 生成预览与人工确认插入
- 一致性检查
- TXT / Markdown / DOCX / PDF / EPUB 导出
- 重新打开恢复

### 6.2 禁做（MVP 非目标）
- 云同步
- 多人协作
- 插件市场
- 导出模板市场与高级排版编辑器（超出基础导出能力）

## 7. 执行流程（每次任务默认流程）
1. 明确目标与交付物。
2. 定义可验证成功标准（命令/测试/行为证据）。
3. 列出假设、歧义与备选方案；必要时先提问。
4. 阅读调用链和入口文件。
5. 做最小可行改动（外科手术式修改）。
6. 执行验证命令并记录结果。
7. 输出：变更、证据、风险、下一步。

## 8. 代码与数据规则
- 前端仅通过 Tauri command 调用业务能力，不新增并行 Node 业务运行时。
- 章节正文必须落盘为 Markdown 文件，数据库仅保存元数据与关联。
- 写入操作优先使用原子写策略（临时文件 + 重命名）。
- 自动保存与正式保存分离，恢复逻辑可验证。
- API Key 不得写入项目目录与日志明文。

## 9. 验证规则
- 行为变更至少提供 1 条可执行验证证据。
- 关键链路（项目/章节/导出）优先覆盖自动化测试。
- 若无法执行某项验证，必须明确说明原因和风险。

## 10. 交付输出格式（默认）
每完成一个模块，按以下结构输出：
1. 实现文件
2. 核心逻辑
3. 测试方式（命令 + 结果）
4. 未完成风险
