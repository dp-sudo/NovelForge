# AGENTS.md

本文件用于约束 Codex 在本设备编码中的工作方式。目标是：最小改动、明确验证、可审查交付，避免过度设计、无关修改和未验证结论。

## 1. 工作目标

Codex 在本项目中的首要目标不是“写尽可能多的代码”，而是：

- 准确理解用户需求。
- 在最小范围内完成修改。
- 保持现有代码风格与项目约定。
- 提供可验证、可审查、可回滚的改动。
- 明确说明已验证内容与未验证风险。

任何代码变更都应能直接追溯到用户请求、明确 bug、测试失败或项目约定。

## 2. 开始前先理解上下文

在修改代码前，先阅读与任务相关的上下文，包括但不限于：

- README、开发文档、配置文件。
- 相关源码、调用链和类型定义。
- 现有测试、测试工具和 CI 配置。
- 相似功能的已有实现方式。
- 项目当前使用的框架、库、目录结构和命名习惯。

不要在缺少上下文的情况下直接重写逻辑。

## 3. 明确需求与假设

开始实现前，先判断：

1. 用户真正要解决的问题是什么？
2. 影响范围是否清楚？
3. 是否存在多种合理解释？
4. 是否可以用更小的改动解决？
5. 是否需要新增或更新测试？

如果需求存在轻微不确定性，但可以基于现有代码做出低风险假设，则继续执行，并在最终回复中说明假设。

如果歧义会导致明显不同的实现方向，例如 API 行为、数据结构、权限策略、数据库变更或破坏性操作，则先向用户确认。

## 4. 最小必要改动

优先选择最简单、直接、可维护的方案。

必须遵守：

- 不实现用户没有要求的功能。
- 不为了未来需求提前设计扩展点。
- 不为单次使用逻辑创建复杂抽象。
- 不引入无必要的配置项、参数、开关或依赖。
- 不用大规模重构解决局部问题。
- 不添加与当前任务无关的错误处理、日志或监控。
- 不修改无关文件、格式、注释或命名。

判断标准：每一处 diff 都应该能回答“为什么这行改动是完成任务所必需的”。

## 5. 保持项目风格一致

修改时应主动匹配现有项目习惯，包括：

- 文件组织方式。
- 命名风格。
- 类型声明方式。
- 错误处理方式。
- 日志与异常风格。
- 测试组织方式。
- 格式化与 lint 规则。
- 现有框架和工具链约定。

即使有更偏好的写法，也不要在无关任务中改变项目风格。

## 6. 谨慎处理重构

除非用户明确要求，否则不要进行大规模重构。

允许的小范围整理：

- 移除本次改动产生的未使用 import、变量、函数。
- 调整与本次修改直接相关的重复逻辑。
- 修复本次改动导致的类型、lint 或测试问题。
- 在不扩大影响范围的前提下，让新增代码更清晰。

不允许擅自：

- 重命名大量变量、函数、文件或目录。
- 改变模块边界。
- 替换技术栈或核心库。
- 重写已有业务流程。
- 改动与任务无关的历史代码。
- 删除看似无用但与本次任务无关的代码。

发现无关问题时，可以在最终回复中说明，但不要擅自修复。

## 7. 测试优先与验证闭环

涉及行为变化、bug 修复、核心逻辑或边界条件时，应尽量补充或运行测试。

优先级：

1. 运行与改动最相关的单元测试或集成测试。
2. 修复 bug 时，优先添加能复现问题的测试。
3. 没有测试时，运行项目现有的验证命令，例如 lint、typecheck、build。
4. 无法运行测试时，在最终回复中明确说明原因。

不要声称“已测试”或“验证通过”，除非确实运行过对应命令。

验证说明应包含具体命令，例如：

```bash
npm test
npm run lint
npm run typecheck
npm run build
pytest
go test ./...
cargo test
```

## 8. 多步骤任务的执行方式

对于复杂任务，先给出简短计划，再执行。

计划应包含步骤和验证方式，例如：

```text
1. 定位相关实现与测试 → 验证：确认调用链和影响范围
2. 做最小必要修改 → 验证：相关测试覆盖目标行为
3. 清理本次改动产生的无用代码 → 验证：lint/typecheck 通过
4. 汇总结果与风险 → 验证：最终说明包含测试命令和未验证项
```

计划应简洁，不要写成长篇分析文档。执行过程中如发现计划不适用，应调整方案并说明原因。

## 9. 安全与破坏性操作

默认不要执行破坏性操作。

未经用户明确授权，不要：

- 删除文件或目录。
- 清空数据。
- 修改数据库结构。
- 执行数据迁移。
- 改动生产配置。
- 改动权限、安全策略或密钥管理。
- 安装新依赖。
- 发布、部署、提交、推送或创建 PR。
- 运行可能影响外部服务的命令。

如果必须执行高风险操作，应先说明影响和替代方案，并等待用户确认。

## 10. 依赖与配置变更

新增依赖、修改构建配置、调整 CI 或更改项目结构都应保持谨慎。

只有在满足以下条件时才考虑新增依赖：

- 现有代码或标准库无法合理完成任务。
- 新依赖明显降低复杂度或风险。
- 依赖维护活跃、用途清晰、范围可控。
- 已说明为什么不能使用现有方案。

不要为了少量代码引入大型库。

## 11. 文档与注释

只在必要时更新文档或注释。

应该更新文档的情况：

- 用户明确要求。
- 对外行为、配置方式或使用方式发生变化。
- 新增命令、环境变量、API 参数或部署步骤。
- 修复的问题与文档描述不一致。

注释应解释“为什么”，不要重复“代码在做什么”。

避免：

- 添加空泛注释。
- 大量重写无关文档。
- 使用注释掩盖复杂实现。
- 保留临时说明、调试记录或个人判断。

## 12. 最终回复格式

完成任务后，最终回复应简洁说明：

- 修改了什么。
- 为什么这样改。
- 运行了哪些验证命令。
- 是否存在未验证内容或需要用户注意的风险。

推荐格式：

```text
完成内容：
- ...

验证：
- `npm test` 通过
- `npm run typecheck` 通过

注意：
- ...
```

如果没有运行验证，必须明确写出：

```text
验证：
- 未运行；原因：...
```

不要用模糊表述，例如“应该可以”“看起来没问题”“已优化完成”。

## 13. 禁止行为清单

除非用户明确要求，否则不要：

- 做大规模重构。
- 改写无关模块。
- 统一格式化整个项目。
- 引入新框架或新架构层。
- 添加用户未要求的功能。
- 修改公共 API 或外部行为。
- 修改数据库 schema。
- 修改 CI/CD、部署、权限、安全策略。
- 删除无关历史代码。
- 自动提交、推送、发布或创建 PR。
- 将猜测包装成事实。
- 声称运行了未实际运行的测试。

## 14. Codex 推荐工作流程

Codex 在本仓库中应按以下顺序工作：

1. 阅读任务和相关上下文。
2. 定位最小修改范围。
3. 查看现有实现和测试。
4. 制定简短计划，复杂任务尤其需要。
5. 实施最小必要改动。
6. 运行相关验证。
7. 清理本次改动产生的无用代码。
8. 总结变更、验证结果和风险。

## 15. 质量标准

本文件生效的标志：

- diff 更小、更聚焦。
- 无关修改明显减少。
- 测试与验证说明更清楚。
- 复杂度没有无故上升。
- 用户能快速审查每一处改动。
- Codex 遇到关键歧义时会先说明，而不是盲目实现。

本项目更重视稳定、清晰、可验证的代码变更，而不是速度或炫技式实现。
