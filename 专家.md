NovelForge — AI 驱动的长篇小说创作辅助软件。

审查结论摘要
项目存在 6 个架构问题，其中：🔴 严重 2 / 🟡 中等 3 / 🟢 建议 1
核心风险：双运行时并行导致的架构漂移 与 AI 命令入口碎片化

必须修复的问题
🔴 严重问题（必须修复）
问题1：旧逻辑残留 + 机制过度设计 + 与主链路冲突
涉及文件：

AGENTS.md:82
src/services/novelforge-mvp.ts:13
src/infra/db.ts:8
tests/integration/runtime-chain-full.test.ts:6
tests/integration/mvp-closed-loop.test.ts:7

问题表现：

仓库保留了一套完整 Node "并行业务后端"（src/services/* + src/infra/db.ts），而主产品链路是 Tauri/Rust
多个 "integration" 测试实际跑的是 Node 业务层，不是 Tauri command 主链路
与仓库自身规则"前端仅通过 Tauri command 调用业务能力"直接冲突

修复要求：

收敛为单一业务后端（Rust/Tauri）作为真源
src/services/* 降级为测试夹具或迁出主目录（例如 tests/dev-engine-fixtures/）
关键集成测试改为调用 src/api/* -> invokeCommand -> Rust command
避免"测得很绿但没测到生产链路"


问题2：AI 命令面碎片化（重复入口）
涉及文件：

src-tauri/src/lib.rs:210, 224
src-tauri/src/commands/ai_commands.rs:140, 363
src-tauri/src/commands/settings_commands.rs:533

问题表现：

同时维护 run_ai_task_pipeline（通用入口）+ 一组 ai_generate_*（按模块入口）+ compatibility-only 命令
同一流水线被多次包装调用，入口增多但能力不增
不同入口返回形态不一致（有的事件链路有 requestId/事件，有的只返回 string），可观测性不统一

修复要求：

统一到 run_ai_task_pipeline(taskType, payload) 一个主入口
ai_generate_* 改为前端 API 层薄封装，不再新增 Rust command
compatibility-only 命令设置明确移除里程碑（版本/日期）并从测试约束中解绑
确保所有 AI 生成链路返回形态统一（统一包含 requestId + 事件机制）


🟡 中等问题（必须修复）
问题3：AiPipelineService 过大，职责耦合过深
涉及文件：

src-tauri/src/services/ai_pipeline_service.rs（1591 行，4 处关键位置：260, 552, 676, 1157, 1260）

问题表现：

单文件混合了：输入校验、上下文收集、路由、Prompt 组装、流式消费、输出归一化、实体解析、落库
任一任务类型改动都可能波及主流程控制，回归面过大

拆分为独立模块：

TaskHandlers/* — 每任务类型解析与落库
PromptResolver — Prompt 组装
PipelineAuditStore — 审计存储


先做"无行为变化拆分"（extract class/module），再逐步精简分支
确保拆分后功能等价，原有测试全部通过


问题4：读接口有写副作用，且默认路由写入路径重复
涉及文件：

src-tauri/src/commands/settings_commands.rs:80, 117, 398, 305

问题表现：

list_task_routes 本应只读，却会触发 upsert_task_route
save_provider 也会补写默认路由，导致"默认路由生成"有两条写路径

修复要求：

把"默认路由初始化"收敛为单一入口（迁移/初始化任务）
list_task_routes 改为纯读，避免调用方产生隐式状态变化
移除或统一 save_provider 中的隐式路由写入逻辑
