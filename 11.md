## 3. Phase 3: 假设与测试

### 3.1 核心假设

**假设 A**: `default_constructed_unit_structs` 可以通过显式字段初始化安全替换，不改变运行时行为。

**验证方法**: 检查 Service 结构体是否有可访问的字段初始化路径。

---

### 3.2 修复策略

#### default_constructed_unit_structs（42 条）

```rust
// 当前模式
let service = TaskHandlersService::default();

// 替换为
let service = TaskHandlersService {
    db: state.db.clone(),
    skill_registry: state.skill_registry.clone(),
    ..Default::default()
};

// 或使用 new() 构造方法
let service = TaskHandlersService::new(db, skill_registry);
```

#### too_many_arguments（2 条）

```rust
// 合并为配置结构体
struct ContextQueryConfig {
    project_root: String,
    character_id: Option<String>,
    state_kind: Option<String>,
    include_derived: bool,
}

fn query_context(config: ContextQueryConfig) -> Result<ContextResponse> { ... }
```

#### manual_range_contains（3 条）

```rust
// 当前
if x >= 0 && x <= 10 { ... }

// 替换
if (0..=10).contains(&x) { ... }
```

## 5. 约束清单

| 约束 | 说明 |
|------|------|
| 不改变公开 API 签名 | 外部调用方不受影响 |
| 不改变功能语义 | 只改写法，不改逻辑 |
| 不批量 suppress | 禁止 `#[allow(clippy::xxx)]` 绕过 |
| 每批配套回归测试 | 测试通过才能合并 |
| 边界逻辑先审核 | range/clamp 类必须验证阈值语义 |
| 假设失败则暂停 | 若显式构造无法实现，回退原设计 |

## 7. 边界检查

| 场景 | 处理规则 |
|------|----------|
| 修复后测试失败 | 回退修改，先补测试再继续 |
| 无法显式构造（私有字段） | 使用 `Default::default()` 并添加 `#[allow(clippy::default_constructed_unit_structs)]` 说明 |
| 语义边界模糊 | 暂停，标注为「需人工审核」，不机械替换 |
| 修复引入新 lint | 同批次修复，不遗留 |
| 假设A验证失败 | 改用渐进式修复（先 suppress 单个，逐步替换） |
