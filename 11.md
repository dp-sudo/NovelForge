更深一层看，这个系统真正要设计的不是几个页面，而是：

AI 如何持续推进生产
系统如何把小说事实、状态、规划组织成可控上下文
用户如何以最低干预成本完成审查、修订、确认

基于你前面的图片、你已经确认的方向、以及当前真实代码，我认为更完整的方案应当这样安排。

1. 核心判断

当前系统如果继续按“蓝图页 + 模块页 + 编辑器”理解，深度还不够。
更准确的理解应当是：

NovelForge 是一个以 AI 为主推进生产力、以故事约束和状态控制保证连续性、以用户审查和修订权作为质量阀门的长篇小说生产系统。

这里最关键的变化是三点：

AI 不是单次生成器，而是持续推进引擎。
用户不是主要生产者，而是方向输入者、审查者、修订者、关键确认者。
系统不是“存资料”，而是“管理故事生产状态”。
2. 当前代码已经说明了什么

从现有代码看，当前项目已经具备一些雏形，但还没有形成真正的“生产系统”。

AI 任务已经统一走 pipeline
证据：
ai_pipeline_service.rs
pipelineApi.ts

技能系统目前本质上是 taskType -> Markdown prompt template
证据：
prompt_resolver.rs
resources/builtin-skills/*.md

编辑器上下文已经能拿到一部分：

蓝图摘要
正式角色
世界规则
剧情节点
前章摘要
正文抽取候选
证据：
context_service.rs
EditorPage.tsx
但当前 prompt 的核心问题是：

拿到的是“静态事实预览”
拿不到足够强的“动态状态”
拿不到明确的“当前卷/当前窗口计划”
拿不到“哪些东西不能漂，哪些东西可以探索”
这就是长篇后期最容易漂的根因。

3. 系统机制不能只做三层，要做“五对象模型”

这版方案里，我建议把系统核心对象明确成 5 类。
这比之前说“第一层、第二层、第三层”更适合指导设计。

3.1 Story Constitution：故事宪法

这是第一层蓝图的升级版，不是普通大纲页。

它只负责保存最稳定、最不该漂的东西：

核心命题
类型策略
世界底层规则
主要人物终局方向
主线终局方向
文风红线
不可违背约束
它回答的问题是：

这本书绝对不能偏到哪里去

所以它不是明细仓库，而是最高约束层。

3.2 Canon Registry：正式资产库

这是当前第二层正式模块的升级定义。

它保存已经正式成立的长期事实：

角色
地点
组织
术语
世界规则
剧情节点
叙事义务
章节计划元数据
正式关系
它回答的问题是：

这个故事里正式存在什么

3.3 State Ledger：动态状态账本

这是当前系统最缺的层，也是长篇质量的关键。

它不记录“人物是谁”，而记录“人物现在怎样了”。

包括：

人物当前身体状态
人物当前情绪状态
人物当前关系温度
人物当前已知信息
当前穿着/伪装/外观变化
当前所在场景
场景当前状态
势力格局当前状态
关键道具当前归属
主线/卷线当前推进状态
未兑现伏笔
当前窗口写作状态
它回答的问题是：

故事现在推进到了哪里

3.4 Execution Workspace：执行工作区

这不是页面，而是 AI 每次生成前临时编译出来的工作上下文。

它由系统自动组装，包含：

故事宪法
当前卷目标
当前窗口规划
相关正式资产
当前动态状态
最近章节承接摘要
本章任务目标
风险提醒
它回答的问题是：

这一次 AI 写作到底应该基于什么来生成

3.5 Review Trail：审查轨迹

这是跨层对象。

它记录：

AI 生成过什么
自动推进了什么
用户改了什么
用户否决了什么
哪些结果被确认升级
哪些规划被修正
它回答的问题是：

为什么系统现在是这个结果

这不是日志装饰，而是未来回退、纠偏、复盘的必要基础。

4. 更深的机制不是“同步”，而是三种动作

你前面一直在说同步。
但更准确的设计语言应该是这三个词：

晋升
回报
编译
4.1 晋升

从低权威内容升级成高权威内容。

例如：

AI 草案 -> 正式蓝图
蓝图条目 -> 正式资产
抽取候选 -> 正式角色/正式关系
状态草案 -> 状态账本正式记录
这不是简单复制，而是“身份升级”。

4.2 回报

从下层运行结果向上层反馈，但不是全文回灌。

例如：

第二层新增了多个宗门，不需要把宗门全文塞回第一层
第一层只需要收到：本卷势力结构复杂度上升，需要补充卷级规划
这才适合长篇。

4.3 编译

写作前，系统不直接把一堆表字段拼进 prompt。
而是按任务类型，把多层数据编译成一份“连续性包”。

这一步极其关键。
当前代码更接近“上下文拼接”，不是“上下文编译”。
这也是后期质量漂移的根因之一。

5. 长篇要稳定，关键不在“更多上下文”，而在“正确类型的上下文”

当前章节生成如果只是拿：

蓝图摘要
角色名
世界规则预览
前章摘要
那对中短篇勉强能跑，但对长篇不够。

真正需要的是 6 类上下文同时存在。

Constitution Context
不能漂的总纲约束
Canon Context
当前章相关正式资产
State Context
当前活跃角色/场景/关系/冲突状态
Promise Context
当前必须兑现的承诺和伏笔
Window Plan Context
当前 10-20 章窗口的局部目标
Recent Continuity Context
最近 1-3 章的承接摘要
这 6 类上下文合在一起，才是高质量长篇生成的真正基础。

6. 第一层蓝图的深度升级：不仅是三层规划，还要加“确定性分区”

前面说过总书级、卷级、窗口级。
更深一层还不够，必须再加“确定性分区”。

我建议蓝图规划里增加 3 个区：

6.1 冻结区

已经写入正文且不可随意漂移
AI 不可自行改写
例如：

已成立的底层规则
已确认的角色核心身份
已发生的重大事实
6.2 承诺区

已经被规划或埋下，后续必须兑现
AI 生成时要强承接
例如：

当前卷目标
高优先伏笔
当前窗口关键节点
6.3 探索区

允许 AI 在边界内扩展
用户可以在这里挑选更优方案
例如：

未来可能的新势力
尚未定稿的支线
后续卷的开放型走向
这比“所有蓝图信息一视同仁”更适合长篇连载。
因为长篇创作天然是：

一部分必须稳定
一部分必须履约
一部分允许探索
7. 技能系统不能再只是平铺任务模板，要升级成“技能编排系统”

你提到要加入：

人物肢体动作
人物着装变化
外貌
情感情绪
背景与场景环境
如果只是把这些都做成一个个独立按钮，那会失败。
原因很简单：

技能会爆炸
任务会碎片化
连续性会更差
用户反而要手动串所有能力
更合理的方式是把技能分成 5 类。

8. 技能系统的五类结构

8.1 Workflow Skills：工作流技能
这是当前系统已经有的主技能类型。

例如：

生成蓝图步骤
生成章节草稿
续写章节
改写选区
创建角色卡
一致性扫描
特点：

面向用户
有明确输入输出
是主要任务入口
8.2 Capability Skills：能力技能
这是你现在提到的那些细节技能的正确归属。

例如：

人物肢体动作编排
人物着装与外貌描写
情绪与心理推进
场景环境渲染
背景信息自然渗透
对话声纹控制
战斗动作与规则兑现
悬疑信息投放
特点：

不一定独立给用户点
更像可注入的能力包
由章节任务按场景类型自动调用
8.3 Extractor Skills：抽取技能
负责把正文中的新增内容沉淀出来。

例如：

新角色抽取
新关系抽取
新场景抽取
新穿着变化抽取
新情绪转折抽取
新状态变化抽取
特点：

写后运行
面向资产和状态更新
8.4 Review Skills：审查技能
负责检测连续性、逻辑性、风格稳定性。

例如：

人物状态连续性审查
服装/外貌连续性审查
情绪弧线连续性审查
场景环境连续性审查
伏笔兑现状态审查
世界规则突破审查
特点：

不直接生成正文
负责发现问题和给修正建议
8.5 Policy Skills：策略技能
负责把项目规则注入 AI 生产过程。

例如：

文风约束
禁用表达
视角限制
术语锁定
设定底线
风格偏好
特点：

常驻启用
作为全局约束存在
9. 你要加的这些技能，正确安排方式不是平铺，而是“技能包”

我建议把你提到的内容整成 4 个核心技能包。

9.1 人物表现包
包含：

肢体动作技能
外貌与面部细节技能
服装与状态变化技能
对话声纹技能
情绪外化技能
作用：

让人物“看得见、听得出、动得对”
9.2 情绪推进包
包含：

当前情绪识别
情绪升级/回落设计
压抑与爆发节奏
情绪与关系联动
内心独白深度控制
作用：

保证人物心理推进连续，不只会写表面动作
9.3 场景环境包
包含：

场景空间构图
光线/声音/气味/温度
时代与背景渗透
环境与情绪联动
场景过渡控制
作用：

让场景不是空舞台，而是有叙事功能的空间
9.4 规则兑现包
包含：

战斗动作逻辑
能力代价与边界
世界规则调用
场景危险性
结果后果追踪
作用：

保证玄幻、科幻、悬疑等类型里不会“规则乱飞”
10. 这些技能不该只影响文本，还要影响数据层

这是更深方案和普通 prompt 增强的最大区别。

你说的技能如果只用来“写得更好看”，价值太小。
它们必须回写数据层。

例如：

10.1 肢体动作/着装/外貌技能
不只影响描写，还应更新：

人物当前外观状态
人物当前着装状态
人物当前伤病/疲劳状态
进入：
State Ledger

10.2 情绪技能
不只影响文笔，还应更新：

人物当前主情绪
压抑情绪
情绪触发源
对某关系的当前温度变化
进入：
State Ledger

10.3 场景环境技能
不只影响环境描写，还应更新：

当前场景状态
当前天气/时段/光照
当前危险等级
当前空间约束
进入：
State Ledger 或 Canon Registry 中的场景对象

10.4 背景渗透技能
不只影响信息表达，还应标记：

哪些设定已经对读者公开
哪些设定仍是隐藏状态
进入：
State Ledger

这样这些技能就不只是写作装饰，而是真正参与连续性控制。

11. 设置页应该怎么安排

结合你给的设置页截图，我建议设置区的职责重新明确。

11.1 模型配置
负责：

可用 provider
模型能力分层
模型池定义
建议不要只看“哪个模型可用”，而要分成能力池：

Planner Pool：规划类
Drafter Pool：写作类
Reviewer Pool：审查类
Extractor Pool：抽取类
State Pool：状态维护类
11.2 任务路由
负责：

taskType -> 模型池
taskType -> 自动化级别
taskType -> 是否自动落草案/落候选/落正式层
taskType -> 需要哪些技能包
这里应该加入一个比现在更强的配置项：

task route + skill bundle + persist policy

11.3 技能管理
不要再只是平铺 Markdown 技能列表。
应支持：

技能分类
技能包
技能优先级
是否常驻
作用范围
自动触发条件
最关键的是加两个概念：

Always On
Conditional On
例如：

术语锁定技能：Always On
情绪推进技能：Conditional On（人物戏重时启用）
战斗动作技能：Conditional On（战斗场景启用）
场景环境技能：Conditional On（环境描写密度高时启用）
11.4 写作风格
这页不能只当作 UI 装饰。
它应该是全局风格向量，供技能包读取。

例如：

语言风格
描写密度
对话比例
节奏
氛围
心理深度
这些值不应只喂给章节主技能。
而应该同时影响：

人物表现包
情绪推进包
场景环境包
背景渗透包
这才会形成真正统一的文风。

12. 更强的运行机制：从“任务执行”升级到“场景感知编排”

这一步是前几版没有说透的。

章节生成不该只是：
chapter.draft + 一份 prompt

而应是：

系统先判断当前章节/当前场景类型
根据类型自动启用技能包
编译上下文
生成计划
生成草稿
运行审查技能
抽取新增状态和资产
例如：

若当前章是“情感对话场景”

启用：人物表现包 + 情绪推进包
降低：规则兑现包权重
若当前章是“战斗推进场景”

启用：动作技能 + 规则兑现包 + 状态更新包
要求回写：伤病、体力、装备、位置
若当前章是“背景展开场景”

启用：场景环境包 + 背景渗透包
要求控制：信息不越权、不暴露未来设定
这比“给 AI 更多说明文字”更强，因为它真正让系统开始理解“当前在写什么类型的场景”。

13. 最终更完备的机制结论

如果把这版更深的理解收束成一句话：

系统机制的核心，不是让 AI 写更多，而是让 AI 在正确的故事权威层、正确的状态层、正确的场景能力包、正确的审查闭环里持续推进生产。

对应到具体安排就是：

用 故事宪法 保证不偏航
用 正式资产库 保证事实成立
用 动态状态账本 保证后续承接
用 执行工作区 保证每次生成拿到正确上下文
用 审查轨迹 保证可回退、可纠偏、可复盘
用 技能包系统 保证 AI 在人物、情绪、场景、规则等维度有可组合能力
用 自动化分级 保证 AI 是主流生产力，用户保留关键审查与修订权
