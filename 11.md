# 代码质量专项修复提示词

> **冲突声明**: 如有冲突，以 `docs/superpowers/specs/2026-04-30-ai-production-system-design.md` 定义为准。

---

## 1. 重复代码（Duplication）

### 问题描述

项目中存在重复逻辑、复制粘贴分支、冗余辅助工具，导致维护成本上升和一致性问题。

### 修复目标

消除重复逻辑，统一实现路径，建立可复用的公共组件。

### 约束

- 抽象层级与项目架构一致
- 不引入不必要的间接层
- 变更后保持功能语义不变

### 验收标准

1. 核心业务逻辑无重复代码块
2. 重复模式统一使用公共函数/组件
3. 消除「复制粘贴后微调」模式

---

## 2. 死代码（Dead Code）

### 问题描述

项目中存在未使用代码、不可达分支、过期标志、调试遗留物，增加维护负担和理解成本。

### 修复目标

清理死代码，还原代码库真实状态，降低维护成本。

### 约束

- 使用静态分析工具定位死代码
- 删除前验证无外部引用
- 保留有历史价值的注释

### 验收标准

1. 无未使用的公共函数/结构体
2. 无不可达的代码分支
3. 无调试语句和临时注释

---

## 3. 不必要的抽象（Unnecessary Abstraction）

### 问题描述

项目中存在透传包装器、投机性间接层、单次使用的辅助层，增加代码跳转成本。

### 修复目标

简化抽象层级，消除无价值间接层，保持代码可读性。

### 约束

- 判断标准：被引用次数 ≤ 1 且无扩展需求 → 展开
- 透传包装器：如果只是转发调用，无额外逻辑 → 内联
- 保持公开 API 稳定性

### 验收标准

1. 无单次使用的 helper 类型/函数
2. 无只转发而不做处理的包装器
3. 抽象层级符合项目架构设计

---

## 4. 边界违规（Boundary Violations）

### 问题描述

项目中存在隐藏耦合、泄漏的职责、错误层的导入或副作用，破坏架构分层。

### 修复目标

修复模块边界，确保分层清晰，职责单一。

### 约束

- UI 层不直接访问数据库
- 服务层不直接操作 UI 状态
- 数据层不包含业务逻辑
- 循环依赖必须消除

### 验收标准

1. 模块依赖关系符合分层架构
2. 无跨层直接调用（除明确接口）
3. 循环依赖清零

---

## 5. 测试缺失（Test Gaps）

### 问题描述

核心行为未锁定、回归覆盖不足、边界情况存在空白，导致修改风险高。

### 修复目标

补充测试覆盖，锁定关键行为，减少回归风险。

### 约束

- 优先覆盖核心业务链路
- 边界条件测试先行
- 不追求 100% 覆盖，追求关键路径覆盖

### 验收标准

1. 核心链路有集成测试覆盖
2. 边界条件有单元测试覆盖
3. 修复后测试不降低

---

## 任务流程

### Step 1: 重复代码检测与修复

**行动**:
- 使用 AST 工具定位重复代码模式
- 识别「复制粘贴后微调」模式
- 统一为公共实现

**约束**: 抽象后保持语义一致

---

### Step 2: 死代码清理

**行动**:
- 运行 dead_code 检测工具
- 验证无外部引用后删除
- 清理调试语句和临时注释

**约束**: 删除前备份，验证不影响编译

---

### Step 3: 不必要抽象简化

**行动**:
- 统计每个类型/函数的引用次数
- 引用次数 ≤ 1 且无扩展需求 → 展开
- 透传包装器 → 内联

**约束**: 不破坏公开 API

---

### Step 4: 边界违规修复

**行动**:
- 分析模块依赖关系图
- 识别跨层依赖和循环依赖
- 重构为符合分层的设计

**约束**: 重构后保持功能不变

---

### Step 5: 测试补齐

**行动**:
- 识别核心业务链路
- 补充边界条件测试
- 验证覆盖率提升

**约束**: 不降低现有测试通过率

---

## 分级问题清单

| 级别 | 问题类型 | 影响 | 优先级 |
|------|---------|------|--------|
| P0 | 循环依赖 | 编译/架构风险 | 必须修复 |
| P0 | 关键链路无测试 | 回归风险 | 必须修复 |
| P1 | 跨层直接调用 | 可维护性 | 尽快修复 |
| P1 | 重复核心逻辑 | 一致性风险 | 尽快修复 |
| P2 | 死代码 | 维护成本 | 计划修复 |
| P2 | 不必要抽象 | 可读性 | 计划修复 |

---

## 约束清单

| 约束 | 说明 |
|------|------|
| 修改前备份 | 每次重构前确认 git 状态 |
| 渐进式修复 | 每次只改一个模式，验证通过后继续 |
| 测试优先 | 关键重构前先补测试 |
| 不可逆判断 | 无法确认安全的修改 → 标注待人工审核 |
